# stashPad 開発ロードマップ

設計の正本である [design.md](design.md) §11 のフェーズ計画を、GitHub issue と紐付けて「何をどの順でやるか」として運用するための文書。issue の起票・クローズや方針変更があったらこの文書も更新する。

## 優先度の考え方

1. **信頼性** — データを壊さない・挙動が裏切らない。ライブラリ(works / タグ / 履歴)は積み上げ資産なので最優先で守る
2. **体験** — 毎日の利用で不満がないこと。「安心して使える」と「満足して使える」は別物
3. **基盤** — 開発・運用が楽であること。CI や lint は急がないが、体験系の改修が増える前に入れると効きが良い
