# stashPad 設計ドキュメント

自宅ファイルサーバ上のメディアデータ(音声・動画・画像/マンガ)を、PC・スマホのブラウザから検索・閲覧・再生できるセルフホスト型メディアライブラリ。

- ステータス: ドラフト(初版)
- 最終更新: 2026-06-10

---

## 1. 背景と目的

DLsite 等で購入したメディアデータがファイルサーバに保管されているが、視聴のたびに PC やスマホへ手動コピーしており不便。これをブラウザから直接ストリーミング視聴できるようにする。

### 解決したいこと

1. **ブラウザからの視聴** — PC・スマホ双方から。音声(マスト)、動画、画像に対応。
2. **タグベースの検索** — DLsite から抽出した作品情報 CSV をインポートして検索に活用。加えて独自タグも付与可能にする。
3. **作品側のフォルダ構造に依存しない** — 作品フォルダの中身を整理し直すことなく、ライブラリ側がエクスプローラ的にフォルダ内をブラウズできる。

### 基本的な利用フロー

```
検索 / 一覧から作品を選択
  → 作品詳細(メタデータ + 作品トップディレクトリのファイル一覧)
  → エクスプローラ的にディレクトリを移動
  → メディアファイルを開くとプレイヤー / ビューアが起動
```

## 2. 要件

### 2.1 機能要件

| # | 要件 | 優先度 |
|---|------|--------|
| F1 | 作品一覧表示(サムネイル付きカード) | MUST |
| F2 | キーワード検索(タイトル・サークル・RJ番号) | MUST |
| F3 | タグによる絞り込み(AND 条件、カテゴリ別ファセット) | MUST |
| F4 | DLsite 作品情報 CSV のインポート | MUST |
| F5 | 独自タグの追加・削除 | MUST |
| F6 | 作品フォルダ内のディレクトリブラウズ | MUST |
| F7 | 音声再生(連続再生・シーク・再生速度変更) | MUST |
| F8 | 画像閲覧(マンガビューア: ページ送り・プリロード) | SHOULD |
| F9 | 動画再生 | SHOULD |
| F10 | ライブラリの自動スキャン(フォルダ → 作品の登録) | MUST |
| F11 | 簡易認証(共有パスワード) | SHOULD |
| F12 | 再生位置の記憶(続きから再生) | LATER |
| F13 | お気に入り・評価 | LATER |
| F14 | zip 等アーカイブのまま展開せず閲覧 | LATER(要検討) |

### 2.2 非機能要件

- **デプロイが簡単**: 自宅サーバ / NAS 上で Docker Compose 一発で起動できること。
- **メディアは読み取り専用**: ライブラリはファイルサーバ上の作品フォルダを一切変更しない。DB・サムネイルキャッシュは別領域に持つ。
- **規模感**: 作品数は数百〜数千程度を想定。全文検索エンジン等の大掛かりな仕組みは不要。
- **シングルユーザー**: 家庭内 LAN(+必要なら VPN 経由)での個人利用。マルチテナントは考えない。
- **モバイル対応**: スマホブラウザでの操作性(レスポンシブ UI、ロック画面での再生コントロール)。

### 2.3 スコープ外(当面)

- トランスコード(非対応フォーマットのリアルタイム変換)— まずはブラウザネイティブ再生可能な形式のみ対応し、必要になったら検討。
- DLsite からの情報自動取得(スクレイピング)— CSV インポートで代替。
- 複数ユーザー・権限管理。

## 3. アーキテクチャ

### 3.1 全体構成

```
┌──────────────┐         ┌─────────────────────────────────┐
│ ブラウザ      │  HTTP   │ stashPad サーバ (Docker)          │
│ (PC/スマホ)  │ ◄─────► │                                   │
└──────────────┘         │  ┌───────────┐  ┌─────────────┐  │
                         │  │ Frontend   │  │ Backend API │  │
                         │  │ React SPA  │  │ FastAPI     │  │
                         │  └───────────┘  └──────┬──────┘  │
                         │                        │          │
                         │       ┌────────────────┼───────┐  │
                         │       ▼                ▼        │  │
                         │  ┌─────────┐   ┌────────────┐  │  │
                         │  │ SQLite   │   │ /media     │◄─┼──── ファイルサーバの
                         │  │ +サムネ  │   │ (read-only)│  │     作品フォルダをマウント
                         │  └─────────┘   └────────────┘  │  │
                         └─────────────────────────────────┘
```

- メディアの実体はファイルサーバ上。stashPad コンテナに **read-only でマウント**する(同一マシンならバインドマウント、別マシンなら NFS/SMB マウント)。
- メタデータ(作品・タグ)は SQLite に保持。サムネイルキャッシュもデータボリューム側に生成。

### 3.2 技術スタック(推奨案)

| レイヤ | 技術 | 選定理由 |
|--------|------|----------|
| バックエンド | Python 3.12 + FastAPI | Range リクエスト対応のストリーミングが容易。メタデータ抽出(mutagen)・サムネイル生成(Pillow)等のライブラリが充実。開発速度が速い |
| DB | SQLite | 単一ファイルでバックアップ容易。想定規模に十分。サーバプロセス不要 |
| フロントエンド | React + TypeScript + Vite | SPA でプレイヤー/ビューアの状態管理がしやすい。エコシステムが厚い |
| デプロイ | Docker Compose | 単一コマンドで起動。メディアのマウントも compose で宣言 |

**検討した代替案**

- **Go 単一バイナリ**: デプロイは最も楽だが、画像・音声まわりのライブラリと開発速度で Python に劣る。
- **Next.js フルスタック**: 1 リポジトリ 1 ランタイムで済むが、大容量ファイルの Range ストリーミングや長時間処理(スキャン)は専用バックエンドの方が素直。
- **PostgreSQL**: 規模に対して過剰。SQLite で困ったら移行(ORM を挟むので移行コストは低い)。

## 4. データモデル

### 4.1 ER 図

```
works ───< work_tags >─── tags
  │
  └───< csv 由来カラムは works に直接保持
```

### 4.2 テーブル定義

**works** — 作品。原則「ライブラリルート直下の 1 フォルダ = 1 作品」。

| カラム | 型 | 説明 |
|--------|----|------|
| id | INTEGER PK | |
| rj_number | TEXT UNIQUE NULL | `RJ01547274` 等。フォルダ名から正規表現で抽出 |
| title | TEXT | CSV があれば CSV のタイトル、なければフォルダ名 |
| circle | TEXT NULL | サークル名 |
| series_name | TEXT NULL | シリーズ名 |
| purchase_date | TEXT NULL | 購入日時 |
| work_type | TEXT NULL | ボイス・ASMR / 動画 / マンガ 等 |
| age_rating | TEXT NULL | 全年齢 / R-15 / R-18 |
| file_format | TEXT NULL | CSV 由来の参考情報 |
| file_size_text | TEXT NULL | CSV 由来の参考情報("1.41GB" 等そのまま) |
| event | TEXT NULL | CSV 由来 |
| root_path | TEXT NULL | ライブラリルートからの相対パス。NULL = CSV のみでフォルダ未発見 |
| thumbnail_path | TEXT NULL | サムネイルキャッシュへのパス |
| created_at / updated_at | TEXT | |

- `root_path` が NULL の作品(CSV にあるがフォルダ未スキャン)は一覧で「未取込」表示。後からフォルダが見つかれば RJ 番号で自動リンク。
- 逆にフォルダはあるが CSV に無い作品は、フォルダ名をタイトルとして登録され、後から CSV インポートでメタデータが補完される。

**tags** — タグ。カテゴリで出自を区別する。

| カラム | 型 | 説明 |
|--------|----|------|
| id | INTEGER PK | |
| name | TEXT | タグ名 |
| category | TEXT | 下表参照 |

UNIQUE(name, category)

| category | 由来 | 例 |
|----------|------|----|
| `genre` | CSV `genres`(カンマ区切り) | ボイス・ASMR, R-15 |
| `detail_genre` | CSV `detail_genres`(スペース区切り) | ASMR, 耳かき, バイノーラル/ダミヘ |
| `voice_actor` | CSV `voice_actor`(`/` 区切り) | 相坂優歌 |
| `scenario` / `illustration` / `music` | CSV 各カラム(`/` 区切り) | カマキリ |
| `custom` | ユーザーが UI から追加 | 任意 |

**work_tags** — 中間テーブル。`(work_id, tag_id)` PK。

**play_states**(Phase 2) — 再生位置記憶。`(work_id, file_path)` ごとに position 秒を保持。

### 4.3 CSV カラムのマッピング

| CSV カラム | 行き先 |
|-----------|--------|
| rj_number, title, series_name, circle, purchase_date, work_type, file_format, file_size, age_rating, event | works の各カラム |
| genres | tags (category=genre)。`,` 区切り |
| detail_genres | tags (category=detail_genre)。空白区切り |
| voice_actor, scenario, illustration, music | tags (各カテゴリ)。`/` 区切り |
| supported_os | 取り込まない(検索価値が低い)|

インポートは **rj_number をキーに upsert**(再インポートで重複しない)。CSV 由来タグは再インポート時に張り直すが、`custom` タグには触らない。

## 5. ライブラリスキャン

### 5.1 作品の検出

1. 設定された**ライブラリルート**(複数指定可)の直下ディレクトリを列挙する。
2. ディレクトリ名から `RJ\d{6,8}` を抽出 → 既存作品(CSV 取込済含む)と突合し、`root_path` をリンク。
3. RJ 番号が取れない場合もフォルダ名をタイトルとして作品登録する(あとで手動編集 / CSV 補完可能)。
4. 各作品についてサムネイル候補を探す: フォルダ直下(→ 浅い階層)の画像のうち、ファイル名に `表紙|cover|jacket|サムネ|main` を含むものを優先、なければ最初の画像。Pillow で縮小キャッシュを生成。

ポイント: **スキャンが見るのはトップディレクトリの対応付けとサムネイルだけ**。フォルダ内部の全ファイルを DB に取り込むことはしない(フォルダ内容はブラウズ時にリアルタイムで読む)。これにより、ファイル構成が変わっても再スキャン不要で、DB が肥大化しない。

### 5.2 トリガー

- 手動: 設定画面の「スキャン」ボタン / `POST /api/scan`
- 起動時: オプションで自動実行
- 定期実行・inotify 監視は LATER

## 6. API 設計

ベースパス `/api`。認証は Cookie セッション(後述)。

### 作品

| Method | Path | 説明 |
|--------|------|------|
| GET | `/works` | 一覧 + 検索。`?q=キーワード&tags=1,2,3&category=...&sort=purchase_date&page=` |
| GET | `/works/{id}` | 詳細(メタデータ + 全タグ) |
| PATCH | `/works/{id}` | タイトル等の手動編集 |
| POST | `/works/{id}/tags` | カスタムタグ追加 `{name}` |
| DELETE | `/works/{id}/tags/{tag_id}` | タグ除去 |

検索仕様: `q` はタイトル・サークル・RJ 番号への部分一致(LIKE)。`tags` は AND 条件。規模的に FTS は不要、必要になったら SQLite FTS5 を追加。

### タグ

| Method | Path | 説明 |
|--------|------|------|
| GET | `/tags?category=&q=` | タグ一覧(作品数付き、ファセット表示用) |

### ファイルブラウズ・配信

| Method | Path | 説明 |
|--------|------|------|
| GET | `/works/{id}/entries?path=` | 指定相対パスのディレクトリ一覧。各エントリに `name, is_dir, size, media_kind` を返す |
| GET | `/works/{id}/file?path=` | ファイル本体の配信。**Range リクエスト対応**(206 Partial Content) |
| GET | `/works/{id}/thumbnail` | サムネイル |

- `media_kind` は拡張子からの分類: `audio | video | image | text | archive | other`
- ディレクトリ一覧は**自然順ソート**(`page2.jpg` < `page10.jpg`)。マンガのページ順に必須。

### 管理

| Method | Path | 説明 |
|--------|------|------|
| POST | `/import/csv` | CSV アップロード → upsert。結果サマリ(新規/更新/リンク数)を返す |
| POST | `/scan` | ライブラリスキャン実行 |
| POST | `/auth/login` / `/auth/logout` | 共有パスワード認証 |

### セキュリティ: パストラバーサル対策

`path` パラメータは必ず以下を通す:

1. 作品の `root_path` と結合して `realpath` 解決
2. 解決後のパスが作品ルート配下であることを検証(symlink で外に出るのも拒否)
3. 違反は 403

## 7. メディア配信とフォーマット対応

### 7.1 配信方式

静的ファイルの **HTTP Range 配信**のみで実現する(トランスコードなし)。音声・動画のシークは Range リクエストで賄える。FastAPI + Starlette の FileResponse ベースで実装。

### 7.2 ブラウザネイティブ対応フォーマット

| 種別 | そのまま再生可 | 備考 |
|------|---------------|------|
| 音声 | mp3, m4a/aac, ogg, opus, flac, wav | DLsite 作品は MP3/WAV 同梱が多く、ほぼ全滅しない。WAV は巨大なので UI 上 MP3 を推奨表示 |
| 動画 | mp4 (H.264/AAC), webm | それ以外(wmv 等)は v1 では「非対応」表示。必要が出たら ffmpeg トランスコードを Phase 3 で検討 |
| 画像 | jpg, png, webp, gif, avif | |
| テキスト | txt (簡易表示) | 添付の台本・README 用 |

## 8. フロントエンド設計

### 8.1 画面構成

```
/                 作品一覧(検索・タグファセット・カードグリッド)
/works/:id        作品詳細(メタデータ・タグ・ファイルブラウザ)
/settings         設定(CSVインポート・スキャン実行)
/login            ログイン
```

### 8.2 作品一覧

- サムネイル付きカードグリッド(スマホ 2 列 / PC 4〜6 列)。
- 上部に検索ボックス + タグフィルタ。選択中タグはチップ表示。
- サイドパネル(スマホではドロワー)にカテゴリ別タグファセット(作品数付き)。

### 8.3 作品詳細 + ファイルブラウザ

- 上部: サムネ・タイトル・サークル・タグ一覧(タップで該当タグ検索へ)・カスタムタグ追加 UI。
- 下部: ファイルブラウザ。パンくずリスト + エントリ一覧。`media_kind` に応じたアイコン。
- ファイルタップ時の挙動:
  - audio → ページ下部に固定のオーディオプレイヤーで再生開始
  - image → フルスクリーン画像ビューアを起動
  - video → 動画プレイヤー(モーダル or インライン)
  - text → テキスト表示

### 8.4 オーディオプレイヤー(最重要コンポーネント)

- 画面下部固定のミニプレイヤー。**ブラウズ操作中も再生継続**(SPA なので自然に実現)。
- 再生キュー = 再生開始したファイルと同じディレクトリ内の音声ファイル(自然順)。連続再生・前後トラック移動。
- シークバー、再生速度(0.5〜2.0x)、音量。
- **Media Session API** 対応: スマホのロック画面・イヤホンのボタンから再生/停止/トラック移動できるようにする(ASMR 用途で実用上ほぼ必須)。

### 8.5 画像ビューア(マンガモード)

- フルスクリーン。同一ディレクトリ内の画像を自然順でページ列として扱う。
- スマホ: スワイプでページ送り。PC: ←→キー / クリック。
- 次ページのプリロード。ページ番号ジャンプ。
- 見開き表示・右綴じ対応は LATER。

## 9. 認証

- 共有パスワード 1 つによるログイン → HttpOnly Cookie セッション。
- パスワードは環境変数で設定。未設定なら認証スキップ(完全に閉じた LAN 用)。
- メディア URL も同一オリジン Cookie で保護されるため、署名付き URL 等は不要。
- 外部公開する場合は VPN(Tailscale 等)経由を前提とし、stashPad 自体を直接インターネットに晒さない運用を推奨(README に明記)。

## 10. デプロイ

```yaml
# docker-compose.yml(イメージ)
services:
  stashpad:
    build: .
    ports:
      - "8080:8080"
    volumes:
      - /mnt/fileserver/dlsite:/media/dlsite:ro   # 作品フォルダ(読み取り専用)
      - ./data:/data                              # SQLite + サムネイルキャッシュ
    environment:
      STASHPAD_LIBRARY_ROOTS: /media/dlsite
      STASHPAD_PASSWORD: ${STASHPAD_PASSWORD:-}
```

- フロントエンドはビルド済み静的ファイルを FastAPI から配信(単一コンテナ・単一ポート)。
- ファイルサーバと別マシンで動かす場合はホスト側で NFS/SMB マウントした上でバインドマウント。
